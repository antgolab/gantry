# /gantry:doubt

在开发过程中主动触发对抗性审查。用全新视角挑战当前实现方案，在问题固化为代码之前发现缺陷。

> 定位：dev 阶段的中途检查点。不是 review 阶段的替代，而是实现过程中的"第二双眼睛"。

## 触发时机

- 做了一个不确定的技术决策
- 感觉"应该没问题"但说不出为什么
- 实现复杂度超出预期
- 偏离了 DESIGN.md 的方案
- 同一个问题改了两次还没解决

## 执行协议

### 1. 准备挑战上下文

收集以下信息交给审查者：
- 当前 git diff（未提交的改动）
- DESIGN.md 相关段落
- REQUIREMENT.md 相关 AC
- 你的决策理由（一句话）

### 2. Spawn 对抗性审查者

审查者必须用**全新上下文**，不继承实现者的推理过程。

审查者的指令：

```
你是对抗性审查者。你的工作是找出当前实现的问题。

假设实现者过度自信。你的目标不是确认方案正确，而是找出它可能错误的地方。

审查维度：
1. 正确性：逻辑是否有漏洞？边界情况是否处理？
2. 简单性：是否有更简单的方案？是否过度设计？
3. 一致性：是否与 DESIGN.md 一致？是否与项目既有模式一致？
4. 完整性：是否遗漏了需求中的某些条件？
5. 风险：最可能出错的地方是什么？

对每个发现，给出：
- severity（critical / major / minor）
- 具体质疑（不是泛泛的"可能有问题"）
- 建议的验证方式（如何证明你的质疑是对的或错的）
```

### 3. 回应挑战

对每个质疑：
- **接受**：修改实现，说明改了什么。
- **反驳**：给出具体证据（不是"我觉得没问题"）。
- **验证**：按建议的验证方式实际验证，贴出结果。

### 4. 循环（最多 3 轮）

- 第 1 轮：初始挑战
- 第 2 轮：对回应的再挑战（如果有未解决的 critical/major）
- 第 3 轮：最终确认或升级

如果 3 轮后仍有未解决的 critical → 停止开发，升级到人工决策。

## 输出

```
## Doubt 审查记录

**触发原因**: [为什么触发 doubt]
**审查轮次**: N/3

### 发现清单

| # | Severity | 质疑 | 回应 | 状态 |
|---|----------|------|------|------|
| 1 | critical | [具体质疑] | [接受/反驳/验证结果] | resolved/open |
| 2 | major | ... | ... | ... |

### 结论

- [ ] 所有 critical 已解决
- [ ] 所有 major 已解决或有明确理由
- [ ] 可以继续开发 / 需要升级决策
```

## 约束

- 审查者**不能**修改代码，只能质疑。
- 实现者**不能**用"时间不够"跳过 critical 质疑。
- 3 轮是硬上限，防止无限循环。
- doubt 记录写入 SUMMARY.md「对抗审查」段。
