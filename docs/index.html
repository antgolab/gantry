核心命题

AI 编程规模化后的
四类系统性失效

大多数 AI 编程工具解决的是能力问题。 Gantry 解决的是上一层：当 AI 已经足够强，工程过程如何保持严谨？

⚡

过程退化

没有外部约束，AI 会自发跳过高摩擦步骤——测试、文档、review——因为生成代码是路径最短的响应。单会话尚可，跨会话后流程遵守率迅速归零。

Process Degradation
🧠

上下文崩溃

超过 50k token 后，AI 开始复读已说过的内容、前后矛盾、对同一段代码给出不同诊断。长上下文不等于好上下文——它是一种慢性失效。

Context Collapse
🌀

范围侵蚀

AI 在局部优化时容易产生"顺手改了"的行为：修一个函数时重构了调用方，删掉"看起来没用"的代码（实际上是反射调用）。累积起来让 PR diff 无从 review。

Scope Drift
📭

知识流失

每个 AI 会话结束，本次踩坑的上下文、技术决策、被否定的方案——全部消失。长期项目里，这是最隐蔽也最昂贵的工程成本。

Knowledge Erosion
Gantry 的应对不是试图让 AI"更小心"，而是 通过工程结构让这四类失效在机制上无法发生。
系统架构

制品驱动的状态机

每次变更被拆解为 9 个有序阶段，每阶段产出一个 .md 制品，下游阶段以上游制品为唯一输入。状态由 CLI 维护，与 AI 工具无关。

制品的价值主张

  • 上下文续接下一个会话的唯一可信输入——无论三天后还是三个月后
  • 决策追溯每一次架构选择都有论证记录，新人入职即可理解历史
  • 知识复用LESSONS.md 沉淀跨任务失败经验，下次碰到同类问题时必检查
  • 范围边界write_files 声明让 diff 超边界行为在提交前可被检测
  • 审计基础完整工程档案支持合规审查、安全评估、代码溯源

适用边界

明确的适用场景

Gantry 为长期、团队、高价值变更设计。对一次性脚本和实验代码，直接让 AI 改更快。

场景 推荐程度 说明
功能改动 > 100 行，需要可追溯制品 ✓ 主力场景 阶段门控 + 制品档案发挥最大价值
团队协作，有 code review 要求，长期维护 ✓ 主力场景 REVIEW 双轮审查 + 角色红线严格分工
接手老项目，需要快速建立 AI 上下文 ✓ 主力场景 context scan Brownfield 护栏专为此设计
个人长期项目，不想每次从头讲上下文 ✓ 推荐 LESSONS + CONTEXT 跨会话续接
改 < 30 行的一次性脚本或简单 bugfix ⚠ 跳过 过程成本大于收益，直接让 AI 改
纯实验性代码、hackathon ⚠ 不适合 流程约束与探索性工作的节奏冲突
快速开始

安装 CLI，初始化目标项目，之后在 IDE 里调用斜杠命令开始第一个变更。

npm install -g @mmmjk/gantry

gantry --version

初始化目标项目

cd your-project

# 自动生成：

# .claude/skills/gantry-*/SKILL.md

设计哲学

六条不可妥协的原则

P1
每阶段一次 fresh context
阶段切换时清窗，靠制品文件传递状态，不靠对话堆叠。
P2
每个产物可被一个文件承载
没有隐式状态，没有"AI 记得"——一切决策可在 .md 里查到。
P3
任务必带 verify 命令
TASKS.md 每项含可执行验证命令，不允许声称"完成"而没跑过。
P4
审查至少两轮
spec 合规 + 代码质量；建议其中一轮跨模型 spot-check。
P5
失败是输入，不是终点
UAT 失败自动产 fix-plan，回到 DEV 循环，最多 3 轮。
P6
Artifact-first，非 Phase-gated
阶段可跳过、可回炉，但制品文件不能缺席。
