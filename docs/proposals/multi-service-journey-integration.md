## 0. 这份文档要解决什么(以及不解决什么)

**核心目的(你的原话拆解):**

1. 对**整个需求结果**负责 ← 根目的
2. 保证**多方在用户路径上的成功和一致** ← 成功标准
3. 涉及**多方的路径测试和问题排查** ← 手段
4. 接口契约一致性 ← 手段之一,不是全部

**第一性结论:根本单位是「用户路径」,不是「接口」。**

- 接口契约一致性是**点**的正确(GetUser 两端字段对齐)。
- 你要的是**链**的正确(下单→支付→扣库存→通知,整条跨服务路径走通)。
- 每段接口都对,整条路径仍可能失败:时序错、状态机不同步、部分成功没补偿、某环超时拖垮全链。这些不是「契约不一致」,是**端到端路径的行为正确性**。
- 「负责」= 路径断了能**定位到哪一方哪一环**,并驱动那一方修到通。

**不解决:** 契约测试(Pact 类)那种可回归、进 CI 的自动化基建。本提案是**流程协议层**的改造,不引入新测试框架。若未来要可回归防护,契约测试可作为正交补充(见 §7)。

## 1. 现状盘点(为什么现在不够)

| 阶段 | 现有能力 | 对「用户路径」的覆盖 |
|---|---|---|
| SPEC (1-requirement) | AC 用 Given/When/Then,已是用户视角 | ✅ 用户视角,❌ 无跨服务维度 |
| DESIGN (2-design) | §2 数据流/架构图(文字版) | ⚠️ 有服务交互的落点,但不强制「路径→服务序列」映射 |
| TEST (5-test) | 第1轮 AC→测试矩阵,按单条 AC | ❌ 按点验证,无端到端路径轮次 |
| INTEGRATION (7) | 全量自动化 + 人工 UAT + 失败诊断 | ⚠️ 是总验收关卡,但偏单体收尾;e2e「如有」可跳;无路径归属机制 |

**缺口:** 没有一条贯穿始终的「用户路径」主线。路径散落在各阶段的隐含理解里,没有显式工件,导致:
- 联调时才临时拼凑「这条路径跨哪些服务」——信息事后重建,易与真实设计脱节。
- 路径断点无归属机制——各服务各说各话,无法「对结果负责」。

## 2. 核心设计:一条「用户路径」主线贯穿三段

引入贯穿工件 **Journey(用户路径)**。它不是新文件,而是**在现有工件里增设结构化段落**,逐阶段精化:

```
SPEC              DESIGN              INTEGRATION
定义路径+成功判据 → 映射服务序列    →  端到端验收 + 断点归属 + 驱动修复
(给"判断成功"依据)  (给"判断归属"依据)  (真正干活:唯一一次端到端验证)
```

**为什么是三段,不是四段(记录决策依据):**

- 贯穿的**唯一目的**是给终点 INTEGRATION 的判定提供**前置事实源**:
  - 判断路径成功 → 需要「成功判据」→ 来自 **SPEC**
  - 判断断点归属 → 需要「每一环该怎么走」→ 来自 **DESIGN**
  - 没有这两个前置定义,归属判定就退化成 agent 拍脑袋。
- 所以 SPEC / DESIGN 是**必要**的(提供依据),但 TEST 阶段**不单独增设端到端轮次**——它会和 INTEGRATION 的路径验收做同一件事(都是拉起多方按路径跑)。端到端验证**只在 INTEGRATION 做一次**,避免重复。
- TEST 阶段专注**单服务**测试;跨服务端到端验证显式移交 INTEGRATION。

### 2.1 SPEC 阶段:定义关键用户路径

在 `SPEC.md` 增设「## 关键用户路径」段。每条路径由**跨服务的 AC 串**组成:

```markdown
## 关键用户路径

### J1:用户下单到支付成功
- 路径 AC:AC-3(下单) → AC-5(锁库存) → AC-7(支付) → AC-9(通知)
- 端到端成功判据:用户看到「支付成功」且订单状态=paid 且库存已扣
- 涉及服务(初判):order / stock / pay / notify
- 失败要求:任一环失败,已扣资源必须回滚(不允许部分成功悬挂)
```

**要点:** 路径由 AC 组合而来,不脱离验收标准。单服务/无跨服务路径的 change,此段写「无跨服务路径」并说明。

### 2.2 DESIGN 阶段:映射路径的服务序列

在 `DESIGN.md §2 数据流` 下,为每条 Journey 补一张**服务时序**:

```markdown
### J1 服务序列
order.CreateOrder
  → stock.Lock (超时 500ms, 幂等键=orderId)
  → pay.Charge (超时 3s, 失败→stock.Unlock 补偿)
  → notify.Send (异步, 失败不阻断主路径)
每一环:输入契约 / 输出契约 / 错误处理 / 超时 / 补偿动作
```

**这里同时覆盖了契约一致性**(每环的输入/输出契约)**和路径行为**(时序/超时/补偿)。契约成为路径的子项,位置正确。

### 2.3 TEST 阶段:不增轮,显式移交

在 `5-test.md` 的范围声明处加一句:**跨服务端到端路径验证移交 INTEGRATION 做,本阶段专注单服务功能/性能/安全等测试**。

理由:端到端路径验证若在 TEST 做一遍、INTEGRATION 再做一遍,是重复(都在拉起多方按路径跑)。只在 INTEGRATION 做一次。TEST 不新增轮次。

### 2.4 INTEGRATION 阶段:路径总验收 + 断点归属 + 驱动修复(唯一一次端到端)

在 `7-integration.md` 增设步骤「跨服务路径总验收」(条件触发:SPEC 有跨服务 Journey 时):

**由单个 Coordinator 角色(Integrator 兼任)执行:**

1. **逐条 Journey 端到端跑** — 按 DESIGN 的服务序列拉起多方,走完整路径
2. **断点定位** — 路径断在哪一环,抓到具体服务 + 具体调用
3. **归属判定(第一性:以已锁工件为唯一事实源)**:
   - 某服务实现**偏离 DESIGN 的服务序列/契约** → 该服务产 `T-FIX-XX`,回退 dev
   - **DESIGN 的路径设计本身有缺陷**(时序错、缺补偿) → 回退 design 重新锁
   - **SPEC 的路径定义/成功判据有歧义** → 回退 requirement
4. **驱动修复、重跑(你的选择:自动定位并驱动到通)**:
   - 责任方修复后,Coordinator **重跑该 Journey**,直到端到端通过
   - R2.6 适用:自动重试 ≤ 3 轮,超限停下要人工决策
5. **产出账本** `JOURNEY-VERIFY.md`:每条 Journey 的验收结果、断点、归属、修复链接、最终状态

**总验收门:** 所有关键 Journey 端到端通过,需求才算完成。这是凌驾于单服务测试之上的最终判据 = 「对整个需求结果负责」。

## 3. 为什么是「单 Coordinator」而不是「多 agent 各扮一端」

(这是前几轮反复纠结的点,结论记录在此)

- **同仓多服务**:Coordinator 能直接读到所有服务的代码/契约,交叉核对即可。
- **多 agent 视角隔离的价值存疑**:两个 agent 是同一模型派生,造不出真人 A/B 之间那种真实认知差;隔离是 prompt 演出来的,抓不到真实分歧,只会抓到静态就能看出的东西。
- **但路径归属需要「隔离的事实源」**——这个隔离不靠 agent 扮演,靠**已锁的 SPEC/DESIGN 工件**。断点判给谁,依据是「谁偏离了已锁工件」,不是「哪个 agent 嗓门大」。这是可靠的、第一性的判定基准。

## 7. 正交补充(本提案之外,供未来参考)

若要「可回归、进 CI」的接口一致性防护,契约测试(consumer-driven, Pact 类)是业界标准解,左移到 dev/test 阶段。它与本提案不冲突:
- 本提案 = **路径行为**的端到端总验收(时序/补偿/结果)
- 契约测试 = **接口点**的可回归防护
两者一链一点,可叠加。本提案不含它。
